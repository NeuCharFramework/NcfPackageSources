# Hotfix - Fix process flow operation conflict error

## 🐛 Problem description

**error message**:
```
Cannot mix synchronous and asynchronous operation on process stream.
```

**Scenario appears**:
This exception is thrown immediately when starting the NCF process, causing the application to fail to start normally.

## 🔍 Cause of the problem

exist`AttachProcessOutputHandlers`In the method, both:
1. `process.StandardOutput.BaseStream.ReadTimeout = 1000`(Synchronous streaming operations)
2. `process.BeginOutputReadLine()`(Asynchronous event handling)

.NET's Process class does not allow mixing synchronous and asynchronous operations on the same stream.

## ✅ Solution

**Modify file**:`Services/NcfService.cs`

**Deleted Code** (lines 950-957):
```csharp
try
{
    // 设置编码为 UTF-8 避免中文乱码
    process.StandardOutput.BaseStream.ReadTimeout = 1000;
    process.StandardError.BaseStream.ReadTimeout = 1000;
}
catch 
{ 
    // 某些进程可能不支持超时设置
}
```

**reason**:
- `ReadTimeout`The setting will mark the stream as synchronous mode
- `BeginOutputReadLine()`Requires pure asynchronous mode
- Conflict between the two causes exceptions

## 📊 Fix verification

Corrected code:
```csharp
private void AttachProcessOutputHandlers(Process? process)
{
    if (process == null) return;
    
    // 直接注册事件处理，不设置 ReadTimeout
    process.OutputDataReceived += (sender, args) => { ... };
    process.ErrorDataReceived += (sender, args) => { ... };
    
    // 开始异步读取
    process.BeginOutputReadLine();
    process.BeginErrorReadLine();
    
    // ...
}
```

## 📝 Lessons learned

### ❌ Wrong approach
```csharp
// 不要这样做！会导致混合操作错误
process.StandardOutput.BaseStream.ReadTimeout = 1000;
process.BeginOutputReadLine(); // ← 冲突！
```

### ✅ Correct approach
```csharp
// 只使用事件驱动的异步方式
process.OutputDataReceived += handler;
process.BeginOutputReadLine(); // ✓ 纯异步
```

### 🎯 KEY POINTS

1. **Choose a method**: either synchronous or asynchronous, no mixing
2. **Event-driven is better**: good real-time performance, no blocking, recommended for CLI output capture
3. **Encoding setting location**: in`ProcessStartInfo`instead of setting it at runtime
4. **ReadTimeout optional**: event-driven approach does not require timeout control
