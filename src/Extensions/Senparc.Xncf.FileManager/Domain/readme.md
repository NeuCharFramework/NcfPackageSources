# Domain layer

The `Domain` folder contains the core business logic and domain models of the system. These models reflect the core concepts and rules of the business, providing the foundation for the entire system. The `Domain` folder typically contains the following types of components:

- Entity: Business objects with unique identifiers whose attributes and behavior change over time.
- Value Object: Business objects without unique identifiers, used to represent characteristics of concepts or entities, with immutable attributes.
- Aggregate: A group of related objects composed of entities and value objects, persisted and transacted as a whole.
- Aggregate Root: The entry point of an aggregate, responsible for maintaining the integrity and consistency of the aggregate.
- Domain Service: Encapsulates business logic operations that typically do not belong to a single entity or value object.
- Domain Event: Represents important business events that occur in the domain, used for decoupling and tracking business changes.

During development, ensure that core business logic and domain models are placed in the `Domain` folder to maintain code clarity and maintainability.

## Domain folder structure

In the `Domain` folder, we create three default subfolders: `Migrations`, `Models`, and `Services`, to organize different types of domain components.

### Migrations folder

The `Migrations` folder stores database migration files related to domain models. These files apply domain model changes to the database to keep the database structure in sync with the domain models.

Typically, this folder is auto-generated by the XncfBuilder module and manual modification is not recommended.

### Models folder

The `Models` folder stores domain model components such as entities, value objects, aggregates, and aggregate roots. These components reflect the core concepts and rules of the business, forming the foundation of the system.

### Services folder

The `Services` folder stores domain services, domain events, and other business logic operations. These operations typically do not belong to a single entity or value object, but involve interactions between multiple domain model components.

Please ensure that during development, different types of domain components are placed in the appropriate subfolders to maintain code clarity and maintainability.
